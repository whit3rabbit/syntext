# Performance Baseline

Baseline captured on 2026-03-26 from the current optimized branch state.

## Environment

- Host: local macOS development machine
- Benchmark harness: Criterion
- Corpus shape: synthetic 300-file repo generated by [`benches/support/mod.rs`](/Users/whit3rabbit/Documents/GitHub/ripline/benches/support/mod.rs)
- Criterion sample size: `10`

## Commands

```sh
cargo bench --bench query_latency -- --sample-size 10
cargo bench --bench index_build -- --sample-size 10
```

## Results

### Query latency

- `query_latency/literal_common`: `3.3948 ms` to `3.4640 ms`
- `query_latency/indexed_regex_rare`: `127.60 us` to `133.67 us`
- `query_latency/full_scan_regex`: `3.4511 ms` to `3.5270 ms`

### Index and commit

- `index_build/full_build_300_files`: `14.186 ms` to `14.688 ms`
- `index_build/commit_batch_single_edit`: `215.90 us` to `236.35 us`

## Interpretation

- Selective indexed regex is much cheaper than broad literal and full-scan regex on the synthetic corpus.
- Single-file incremental commit is currently sub-millisecond on this corpus.
- Future commit-path work should compare against `commit_batch_single_edit`.

## Post Stable-File-ID Rerun

After the stable file ID refactor, the same commands were rerun sequentially,
one benchmark at a time, to avoid cross-benchmark contention.

### Query latency

- `query_latency/literal_common`: `3.1894 ms` to `3.2895 ms`
- `query_latency/indexed_regex_rare`: `144.40 us` to `149.98 us`
- `query_latency/full_scan_regex`: `3.3969 ms` to `3.4613 ms`

### Index and commit

- `index_build/full_build_300_files`: `15.336 ms` to `15.841 ms`
- `index_build/commit_batch_single_edit`: `340.01 us` to `408.55 us`

### Comparison

- Query latency is effectively unchanged.
- Full build is close to baseline and within the same order of magnitude.
- `commit_batch_single_edit` regressed materially versus the baseline
  `215.90 us` to `236.35 us`.

That means the stable file ID groundwork is functionally correct, but it did
not pay for itself yet on the incremental commit path.

## Post Incremental Bitmap Maintenance Rerun

After making `PathIndex::build_incremental(...)` update extension and path
component bitmaps in place instead of rebuilding them from the full visible
path set, the index-build benchmark was rerun sequentially.

### Index and commit

- `index_build/full_build_300_files`: `15.759 ms` to `16.360 ms`
- `index_build/commit_batch_single_edit`: `134.96 us` to `139.23 us`

### Comparison

- Full build remains slightly slower than the original baseline, which is
  expected because stable file ID bookkeeping still adds some overhead.
- `commit_batch_single_edit` improved sharply versus the regressed stable-ID
  state (`340.01 us` to `408.55 us`), and now beats the original baseline
  (`215.90 us` to `236.35 us`).

This is the first measurement showing the stable file ID direction paying off
on the commit path. The long-term design still needs more data on larger
corpora, but the single-edit incremental case is now clearly better than both
previous measured states.
